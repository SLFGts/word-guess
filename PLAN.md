# 猜词小游戏 — 从0到1实施计划

## Context（背景与目标）

灵感来自抖音直播里有人玩的猜词游戏（国外同类为 Contexto / Semantle）。核心机制：系统随机给定一个目标词（中文，≤5字），玩家盲猜任意词，系统返回该词与目标词的**相关度百分比**，引导玩家不断逼近直至猜中；玩家可主动依次查看提示辅助猜中。

这是一款**天生适合 AI 技术落地**的游戏——相关度本质上是词向量语义相似度。最终以**微信小程序**形态上线，定位多人聚会聊天休闲场景，强调竞技性与社交趣味。

**本次计划聚焦 P0 MVP**：先用单人闯关模式跑通"猜词→相似度反馈→提示→猜中"的核心闭环，验证玩法是否有趣、分数体验是否合理。多人在线、社交分享、商业化均放到后续阶段。技术方案基于已确认的决策：后端走"云开发起步→P1 迁 Python"、P0 先单人、词向量用"腾讯 AI Lab v0.2.0 词级向量主力 + FastText subword 兜底"、提示三级主动查看、暂不变现。

预期结果：2-3 周产出一个可在微信开发者工具真机预览、核心玩法完整可玩的小程序 MVP。

## 一、核心算法链路（相似度 → 0-100 分）

采用**词级词向量余弦相似度 + 排名映射**（参考 Contexto 用 GloVe、Semantle 用 Word2Vec 的先例——词级共现向量对"单词联想"比句子级 embedding 更对口）：

```
离线提取（一次性，本地 Python 脚本）：
  下载腾讯词向量 d200-v0.2.0-s.tar.gz（1.54GB，200维，语料截止 2021年3月）
  → gensim 加载 → 三层过滤（正则1-5字纯中文 + jieba词性n/v/a/i + 白名单∩腾讯表全交集）
  → 白名单 = Rime高频内容词 + 成语(chinese-xinhua) + 手工新词；排除停用词+政治词
  → pypinyin 算字数/首字/拼音首字母 → 提取向量
  → 产出 data/vocab-final.json（词+结构属性）+ data/vectors.json（向量）
  （答案池 53219 词，含成语+常用词+新词+单字+网络生活词，每个词都有向量，零 OOV）

运行时（云函数 guess）：
  1. 开局选定目标词 T，取其向量 V_T
  2. 计算 V_T 与答案池全部词向量的余弦相似度，降序排序 → 得到「词→排名」表
     （53219 词 NumPy 矩阵运算约几十 ms，无需向量数据库）
  3. 玩家输入猜测词 G：
     a. 若 G 不在答案池 → 返回"词库里没有这个词"（不计次数）
     b. 查 G 的排名 rank
     c. score = round( (1 - log(rank)/log(N)) × 100, 2 )   // 对数排名映射，保留两位小数（线性下top20全100无区分度，对数拉开头部梯度；尾部相邻rank整数分易同分，两位小数拉开：rank2→93.6/10→78.8/100→57.7/1000→36.5）
     d. G == T → rank=1 → score = 100，游戏胜利
```

> 选对数排名映射而非线性映射的原因：余弦相似度分布偏态严重，线性排名映射下 top20 词全挤在 100 分（rank 500 仍 98 分）完全无区分度（试玩实测）；对数映射压缩尾部无关词区间、放大头部差异，逼近过程有持续正反馈。参考 Contexto 用纯排名数字、Semantle 用 cos×100+Top1000。

> 选词级向量（腾讯 Word2Vec/Directional SG）而非句子级 embedding（千问 v4/BGE）的原因：词级向量训练目标是"词-词共现"（SGNS 隐式分解 PMI 矩阵），恰好等价于猜词游戏要度量的"词-词相关度"，目标函数天然对齐；句子级 embedding 优化"句子-句子匹配"，对单词细粒度联想是副产物，且单词向量被句子上下文调制、不一致。Contexto/Semantle 的成功即基于词级共现向量。

**关于词库规模的成本/延迟**：离线一次性提取（5 万词也就几秒 NumPy）；运行时选定目标词后用矩阵运算一次算出相似度并排序（5 万词约几十 ms）。扩大词库不增加运行时成本，唯一约束是 vectors.json 内存占用——200 维下 5 万词仅 ~40MB，float16 再减半，云函数内存无压力。

**新词覆盖说明**：腾讯 v0.2.0 语料截止 2021 年 3 月，能原生覆盖 2020-2021 走红词（内卷、躺平）；2022+ 新词（显眼包、city不city）不在表 → MVP 答案池只选在表的词（不刻意塞新词，回避 OOV）；V1.1 起用 FastText cc.zh.300.bin 的 subword 给新词合成向量兜底（质量中等，仅作 fallback）。

| 层 | 选型 | 说明 |

| 前端 | 微信小程序原生（WXML/WXSS/JS） | 文字 UI 友好；若审核归类为游戏需迁小游戏形态（见合规说明） |

| 持久化 | wx.setStorageSync（本地） + 云数据库（历史/category 缓存） | 单人模式状态走本地 |

## 词库设计：多元化与主题化（趣味性关键）

纯常规名词（苹果、香蕉）易枯燥，聚会场景需"梗"与新鲜感。答案池三层过滤后含 1-5 字词（含单字），天然多元层级：

| 层级 | 示例 | 说明 |
|---|---|---|
| 具象名词 | 苹果、熊猫、地铁 | 简单档，新手友好 |
| 抽象概念/情绪 | 内卷、躺平、emo、社恐、松弛感 | 聚会共鸣点（腾讯2021语料覆盖内卷/躺平） |
| 成语/俗语 | 画蛇添足、对牛弹琴 | 文化档，难度适中 |
| 网络热词/梗 | 显眼包、city不city | 传播档（2022+新词不在腾讯表，V1.1 用 FastText 补） |
| 影视/综艺/人物 | 狂飙、甄嬛、孙悟空 | 流行档 |
| 节日/季节应景 | 中秋、年终奖 | 时效档，常玩常新 |

配套机制：
- **主题专场**：聚会开局可选"热词专场""成语专场""影视专场"，按 category 筛答案池
- **难度分级**：简单（具象）/ 普通（混合）/ 困难（抽象 + 生僻）
- **持续更新**：定期补充新热词（V1.1 起 FastText 兜底新词向量）
- 多元化词库也是 P1 在线对战的核心谈资来源

## 提示系统设计（三级：描述/锚点/提问裁判，均 LLM 生成；玩家主动依次查看）

不按时间触发，提示主动权交给玩家。三级按信息量由低到高依次解锁（看 T2 需先看 T1；提问裁判需先看 T1+T2），已解锁提示常驻显示。**每局每级只可点一次**（T1/T2 点击后按钮变暗；提问裁判每局 1 次提问机会），开新局重置。

**字数信息前置**：开局 `/new_game` 默认返回目标词字数（原 T2 结构提示的字数部分下放到开局基础信息），不再占一级提示。

| 级别 | 提示内容（示例：目标词"猫") | 生成方式 | 运行时 LLM? | 延迟 |
|---|---|---|---|---|
| **T1 描述** | "一个名词，和小动物有关" | 每局点击时实时调 qwen-turbo 生成（不缓存） | 是，每局一次 | 0.9-2s |
| **T2 锚点** | "和「狗」有关联" | 每局点击时实时调 qwen-turbo 生成联想方向词 | 是，每局一次 | 0.9-2s |
| **T3 提问裁判** | 玩家问"它是动物吗"→"是的" | 玩家输入问题，LLM 结合 T1/T2+目标词作答 | 是，每局 1 次 | 0.9-2s |

设计要点：
- **T1 去缓存每局新生成**：不沿用首次结果，保证不同局提示新颖；代价是放弃 P1 多人模式"同目标词提示一致"的竞技公平性（P0 单人无此约束，P1 再评估）
- **T2 改 LLM 生成联想方向词**：原 T3 词向量 most_similar top1 实测天然偏向同词根变体（猫→猫猫、内卷→内卷化、孙悟空→悟空），等于换皮答案且恒≈94分送分过猛；改用 LLM 生成"相关但非同词根/变体"的联想词，更像人脑联想。T2 给的词不在答案池也行——只作方向提示，玩家参考方向自己另猜
- **放弃同源/确定性原则（已知取舍）**：T2 改 LLM 后，"与分数体系同源、确定性、竞技公平"三条原设计原则不再成立，换得联想质量提升。P0 单人可接受，P1 多人竞技阶段需重新评估是否回退确定性方案
- **无 key 降级**：未配 LLM_API_KEY 时，T2 降级为词向量过滤同词根后取首个近邻（仍优于原 most_similar top1 的变体问题）
- **每局每级一次**：T1/T2 点击后按钮变暗禁用，防重复调用刷提示、控成本
- **T3 提问裁判（交互式）**：玩家输入一个问题，LLM 作裁判结合 T1描述+T2锚点+目标词作引导性回答。约束：只答与已揭示提示方向相关的问题（无关问题如天气/闲聊回固定拒绝语"请围绕已给出的提示方向提问"）、禁直接索答（回"不能直接告诉你答案，请继续猜"）、不出现目标词字、方向性问题可答是/否。须先解锁 T1+T2，每局 1 次提问（无对话历史）。相关性/禁索答靠 few-shot 示例强化（实测天气+索答 100% 拦截），但 LLM 软约束非硬保证
- **emoji 不进提示链**：改作 UI 装饰和结算页点缀
- **计分代价**：MVP 不设（休闲优先），P2 竞技阶段再加入"用提示扣分"机制

> **设计演变**（2026-07-08，试玩反馈驱动）：原三级设计（T1 类别 LLM 缓存 / T2 结构词向量 / T3 锚点词向量）。试玩暴露三问题：①T1 缓存致首次不佳提示一直沿用、损新颖；②T2 首字母拼音歧义大、提示弱；③T3 词向量近邻偏向同词根变体、且恒≈94 分送分过猛。遂改为两级均 LLM：T1 去缓存每局新生成、T2 结构级删除（字数前置开局）、原 T3 改 LLM 生成联想方向词（不在答案池也行）。后又新增第三级「提问裁判」：玩家向 LLM 提问获取线索（须先解锁 T1+T2、每局 1 次、无对话历史），裁判结合已揭示提示作引导性回答，靠 few-shot 强化相关性/禁索答约束。

> **2026-07-10 演进**：①开局选题从纯随机改 **smart_pick**——抽5候选→LLM一次调用选最适合出题词+预生成T1/T2（结构化JSON+response_format校验：chosen必在候选否则重试/泄露置None走兜底）→存后端缓存，玩家点提示秒返(~0.002s)；预生成失败走原gen_t1/gen_t3函数兜底，无key/全失败走纯随机。实测预生成命中率83-100%、真剧透泄露0。选词prompt含回避约束（生僻/繁体字、生僻成语、口语化短语、专业术语）实测生效。②泄露判定从"不含任何字"放宽为**"不含2字及以上子串"**：含常用字target（水缸/电视机/玉龙雪山）描述必撞单字，严格判定致预生成50%失败走兜底；放宽后单字（水/电/鱼）不剧透放过、2字子串（雪山/经济）才拦，预生成→100%命中。prompt软约束仍写"不得出现目标词任何字"（尽量让LLM避字），硬校验兜底放过避不开的单字。

### 阶段 A — 核心算法验证（无微信依赖，立刻可做）

**Step A1 — 下载词向量 + 配 LLM Key（Day 1）**
- 下载腾讯词向量 d200-v0.2.0-s.tar.gz（1.54GB）：`https://ailab.tencent.com/ailab/nlp/en/data/tencent-ailab-embedding-zh-d200-v0.2.0-s.tar.gz`
- 解压到 `data/raw/`，得 .txt 词向量文件
- 在 `.env` 配置 qwen-turbo（阿里云百炼 OpenAI 兼容）：`LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL=qwen-turbo`
- 微信小游戏账号、开发者工具、云开发均暂不需要

**Step A2 — 词向量质量实测（Day 1，定生死）**
- `scripts/eval_vectors.py`：gensim 加载腾讯词向量，查"内卷""躺平""显眼包""苹果"在不在表、`most_similar` 近义词是否符合直觉（"内卷"→竞争/996/卷 而非学术义）
- 5 分钟跑完，直接验证腾讯 v0.2.0 是否胜任主力；不通过则评估改用 Chinese-Word-Vectors 或混合

**Step A3 — 答案池与向量提取（Day 2）**
- `scripts/build_vocab_and_vectors.py`：加载腾讯词向量 → 筛 2-5 字纯中文词（去标点/英文/数字/单字）→ 取高频前 3000-5000 → pypinyin 算字数/首字/拼音首字母 → 提取向量 → 产出 `data/vocab-final.json` + `data/vectors.json`
- 抽查：词对余弦相似度合理性（苹果↔香蕉 > 苹果↔汽车）

**Step A4 — 本地玩法原型（Day 3-4，强烈推荐）**
- `scripts/serve_local.py`：Python 起本地 HTTP 服务，加载 vectors.json，实现"选目标词→算排名表→输入猜测词→返回 0-100 分"完整链路
- 配极简 HTML 页（浏览器打开即可）：输入框 + 分数显示 + 猜测历史 + 三级提示按钮
- **目的**：投入小程序开发前，用最低成本验证核心玩法是否真的有趣、分数梯度是否合理——产品最大风险点应最早验证
- 通过则进入阶段 B；不通过则回头调答案池/映射函数/提示设计

### 阶段 B — 小程序集成（需微信环境，阶段 A 验证通过后进入）

**Step B1 — 微信环境准备（Day 5）**
- 注册微信小游戏账号，获取 AppID（注册基本即时，无需等审批）
- 下载微信开发者工具
- 开通微信云开发，记下环境 ID
- 资质合规暂不处理——开发、真机预览、内测均无需资质（见文末合规说明）

**Step B2 — 云函数后端（Day 5-7）**
- `cloudfunctions/newGame`：随机选目标词、生成 gameId、预生成 T1 category（查云数据库缓存→无则调 qwen-turbo 生成并写缓存）
- `cloudfunctions/guess`：加载 vectors.json → 算余弦相似度 → 排名 → 映射 0-100 → 返回分数；处理 OOV（答案池外词提示"不在词库"）
- `cloudfunctions/getHint`：T1 返回 newGame 预生成的 category；T3 用词向量算目标词 most_similar top1 返回
- 云开发 CLI 本地调试（算法逻辑直接复用阶段 A 的 Python 实现，翻译成 JS）

**Step B3 — 前端游戏核心页（Day 8-12）**
- `pages/game`：输入框、大字分数显示、猜测历史（按分数降序）、三级提示面板
- `components/score-bar`：冷→热渐变 + 数字滚动动效
- `components/guess-list`：历史卡片，高分高亮
- `components/hint-panel`：三个提示槽，玩家点击依次解锁（看 T2 需先看 T1）
- 接通云函数，跑通"猜词→分数→展示"闭环

**Step B4 — 提示与结算（Day 13-15）**
- 提示面板交互：玩家主动点击依次解锁 T1→T2→T3，已解锁提示常驻显示
- `pages/result`：猜中后展示用时/次数/分数曲线、emoji 庆祝点缀，"再玩一局"
- `pages/profile`：历史成绩（云数据库）

**Step B5 — 体验调校与打磨（Day 16-18）**
- 挑 10-20 个不同难度目标词实测，校验分数梯度，必要时调映射函数
- 调提示梯度（三级信息量是否合理）、UI 微动效、音效、空状态文案

## 六、关键技术难点与应对

| 难点 | 应对 |
|---|---|
| 词级向量语料旧 | 腾讯 v0.2.0 语料 2021.3，覆盖 2020-2021 梗词；2022+ 新词 V1.1 用 FastText subword 兜底 |
| 短词向量区分度 | 腾讯 Directional Skip-Gram 词级向量对短词天然友好；答案池筛 2-5 字高频词 |
| 相似度分布偏态 | 用对数排名映射而非线性映射（线性下 top20 全100无区分度，对数拉开头部梯度：rank2→93/10→77/100→55/1000→32） |
| OOV（玩家输入答案池外词） | 提示"不在词库"不计次数；P1 起对高频输入词动态入库 |
| T1 提示生成延迟 | 开局预生成+云数据库缓存，玩家点击零延迟 |
| 云函数冷启动/向量加载 | vectors.json 106MB（53219×200），原型本地加载约15秒；小程序需 float16压缩/分片/CDN（B2 处理） |
| 腾讯词向量 "research only" 许可 | 技术上最优但商用有声明风险，上线前重新评估（联系授权或届时切 Chinese-Word-Vectors Apache-2.0） |
| **词库质量未达预期（2026-07-08 归档，暂不换）** | 腾讯 d200-s 实测不足：语料截止 2021.3 无 2022+ 新词、抽象/梗词字面义错位（内卷→蓬松/卷曲、躺平→侧卧）、部分近邻不符直觉。**暂不换**。2026-07-09 已实测 C-WV Mixed-large(300维,Apache-2.0,语料2017-18)并排对比腾讯d200-s：C-WV语料更旧致社恐/扫码/松弛感不在词表、996当数字串、内卷/躺平无社会义，对聚会场景比腾讯的同词根变体问题更致命；C-WV仅具象联想略好(猫→猫咪/狗)但T2已改LLM用不上。**结论维持腾讯主力、C-WV归档**。BGE-M3/M3E后续视需要再测。注：LLM 联想图只能增强提示层、不能替代分数体系（5万×5万词对 LLM 只覆盖每词 top-K，猜非联想词无分数）；自训练近年语料词向量为 P3 方案 |
| 成本 | 词向量离线提取零成本；运行时仅 T1 首次调 qwen-turbo（几分钱级）；云开发免费额度覆盖 MVP |

## 七、上线资质与合规（仅正式上线前处理）

**开发阶段（P0-P1）零资质需求**：本地开发、真机预览、小范围内测分享都不需要软著/备案/版号。先把产品做出来验证玩法，合规是上线前的事。

**正式上线前需核实与办理**（政策易变，以微信公众平台当前公告为准）：
- 游戏类内容必须以**微信小游戏**形态上线，不能走普通小程序规避——猜词游戏有得分/竞技机制，审核可能要求归为游戏类，届时需迁小游戏形态
- 个人主体能否上线小游戏**当前政策有不确定性**（一说免费休闲游戏可免版号、一说游戏类目须企业主体）——上线前务必去 mp.weixin.qq.com 核实
- 常见所需资质：ICP 备案、软件著作权、游戏自审报告；无内购时通常无需游戏版号
- **若个人主体走不通**：注册个体工商户（成本几百元、流程简单）是最务实转身路径；或先以 H5/抖音小程序验证传播再回迁
- 软著本身不贵（自助 270-300 元，电子认证 ~600 元/10-15 天，代办 800-1500），贵在周期（自助 2-4 个月），确定上线后提前启动即可
- 暂不变现 → 不接广告、不开内购，降低合规门槛

**结论**：P0 阶段不被资质卡住，专注产品；上线前 1-2 个月再核实政策、并行办软著与备案。

## 八、验证方式（端到端）

1. **词向量质量**：跑 `eval_vectors.py`，查"内卷/躺平/苹果"在不在腾讯表、`most_similar` 近义词符合直觉（定生死，A2 必过）
2. **答案池与向量**：跑 `build_vocab_and_vectors.py`，`vocab-final.json` + `vectors.json` 生成；抽查词对余弦相似度合理
3. **云函数**：云开发 CLI 调试 `guess`，输入猜测词返回 0-100 分且梯度合理、OOV 拦截；`newGame` 预生成 category、`getHint` 返回三级提示
4. **前端闭环**：微信开发者工具真机预览，完整走"开局→猜若干词→看分数逼近→主动点提示依次解锁→猜中→结算"
5. **体验调校**：找 3-5 个目标词实测，确认①分数随逼近单调上升、②无关词稳定低分、③三级提示梯度合理、④UI 动效流畅
6. **多人预留**：单人模式状态管理设计成可扩展（本地 store 留出多人同步接口位），为 P1 在线房间铺路
